# ADR-0021: Scheme guidance RAG, text-chat fallback, and the real assistant frontend

- **Status:** Accepted
- **Date:** 2026-07-20
- **Sprint:** Sprint 2 (T12)

## Context

T12 asks for scheme guidance (RAG over government scheme content) and a real assistant frontend, replacing the T04 wireframe and T10/T11's audio-only backend with something an SHG member can actually use: voice, text, citations, and a low-bandwidth path.

## Decision

**RAG split: ml-services retrieves, voice-service grounds and answers.** A new `scheme_chunks` pgvector table (migration `20260720000000_scheme_chunks`, `vector(384)` — matching T08's actual `paraphrase-multilingual-MiniLM-L12-v2` model, deliberately not the 768-dim placeholder on `Product`/`Buyer`) holds a small, hand-curated, honestly-cited set of real AP/MEPMA SHG credit schemes (DAY-NULM, Vaddi Leni Runaalu, PM SVANidhi, SthreeNidhi, PM Mudra Yojana, MEPMA's 2025-26 action plan — see `apps/ml-services/app/scheme_guidance/content.py` for the sourced text and real URLs). `ml-services` exposes `POST /scheme-guidance/search` (embed query, cosine-similarity search, filter below a relevance floor, return ranked chunks with citations) — pure retrieval, no LLM call. `voice-service`'s new `scheme_guidance` tool calls that endpoint and hands the chunks to the same Groq LLM already driving the conversation, with a system-prompt instruction to answer _only_ from the retrieved chunks, cite the source, and admit "no match" rather than fall back to general knowledge. This keeps retrieval and generation in the services that already own embeddings (ml-services, ADR-0017) and conversation (voice-service, ADR-0019), instead of adding a third service.

**No vector index on `scheme_chunks`, unlike `Product`/`Buyer`'s `ivfflat` indexes.** Verified empirically, not assumed: with an `ivfflat` index (even the same conservative `lists = 100` used for the other tables) on a table this small (a few dozen curated chunks), many query vectors' nearest cluster held zero of the true matching rows, and the query silently returned nothing for a clearly relevant question ("What loans are available to street vendors?" scored 0 results with the index, 0.543 top score without it). `ivfflat`'s approximate-search tradeoff only pays off at real scale; for a table that will only ever hold a curated, human-sized document set, a sequential scan is both correct and fast. Confirmed the `MINIMUM_RELEVANCE_SCORE = 0.3` floor (reused from `categorization`'s own empirically-tuned value, ADR-0017) separates real matches (0.53-0.76 in testing) from off-topic queries (0.13-0.20) cleanly on this corpus.

**Text-chat fallback is a dedicated stateless HTTP endpoint (`POST /sessions/{session_id}/message`), not RTVI client-message injection.** `usePipecatConversation`'s `injectMessage` (the obvious-looking alternative) was inspected in the installed `@pipecat-ai/client-react` source and confirmed to be purely client-side UI state — it never reaches the server, so it cannot substitute for actually sending a user's typed message into the LLM's conversation. The new endpoint reuses `build_tools()`, the shared system prompt (extracted to `app/prompts.py` so voice and text stay behaviorally identical), and the same Redis-backed `SessionStore`/`session_id` as the live call, so a member can switch between speaking and typing mid-conversation without losing context. Since there's no Pipecat pipeline for a single HTTP request/response, tool-calling against Groq is done directly via the `groq` SDK (already a dependency) with a hand-written `FunctionSchema` → OpenAI-tool-format conversion, and a minimal stand-in for `FunctionCallParams` (`_CapturedToolCall`, exposing just `.arguments`/`.result_callback`, the only two attributes any handler in `app/actions.py` actually touches) — simpler and more transparent than pulling Pipecat's adapter machinery into a code path with no audio at all.

**Frontend uses the official `@pipecat-ai/client-js` / `@pipecat-ai/client-react` / `@pipecat-ai/small-webrtc-transport` packages**, not hand-rolled WebRTC/RTVI. `VoiceVisualizer` (official waveform component) and `PipecatClientAudio` (automatic TTS playback) meant no custom `AnalyserNode`/canvas code was needed. `usePipecatConversation`'s `messages` (including `role: "function_call"` entries) is the single source of truth for the transcript, rendering both live voice turns and injected text-chat turns through the same list — action results from both channels render through one `ActionResultCard` component that switches on the result's own shape (not the tool name), so it degrades gracefully if a handler's result shape changes.

**Low-bandwidth mode disables the voice call entirely (forces text-only), rather than only turning off waveform animation.** Audio streaming is the actual bandwidth cost; a cosmetic-only toggle wouldn't address what "low-bandwidth" is asking for. Toggling it on while a call is active disconnects it.

**A real, self-introduced wire-format bug was found and fixed before any frontend code was written against it**, by installing the actual npm packages into a scratch directory and reading their bundled source rather than trusting docs: `/api/offer` expected `pcId`/flat `sessionId`/`accessToken`/`language`, but the real client sends `pc_id` (snake_case) with custom connect params nested under a `requestData` key. Fixed in `apps/voice-service/app/main.py`.

## Alternatives Considered

- **A separate `scheme-guidance-service`** — rejected; ml-services already owns embeddings and pgvector search (T08), voice-service already owns the conversation and Groq client (T10) — a third service would duplicate both for no benefit at this scale.
- **RTVI client-message injection for text input** — rejected after confirming (by reading the actual installed source, not the docs) that `injectMessage` never reaches the server.
- **Reusing Pipecat's pipeline/adapter machinery for the text-chat endpoint** — rejected as unnecessary complexity for a single request/response call with no audio; a direct Groq SDK call with a small hand-written tool-schema converter is simpler and easier to reason about.
- **Keeping the `ivfflat` index "for consistency" with `products`/`buyers`** — rejected once the empty-cluster false-negative was reproduced; consistency isn't worth a search that silently misses real matches.
- **A comprehensive scheme knowledge base** — rejected in favor of a small, honestly-cited set (6 real schemes, real source URLs). Broader coverage is future work, not a POC blocker; the assistant is instructed to say "no match" rather than guess when a question falls outside this set.

## Consequences

- Positive: retrieval is grounded and citable (every answer traces to a real scheme name + source URL); voice and text conversations share one session, one tool set, and one system prompt, so behavior doesn't drift between channels; the `ivfflat`-index finding is a real, reusable lesson for any future small pgvector table in this project (don't default to the large-table index just because a bigger sibling table has one).
- Trade-offs: the scheme content set is small and non-exhaustive — questions about schemes outside the curated six are correctly declined, not answered, which is honest but limited; the production build's main JS bundle is now ~1.24 MB (up from including the Pipecat client SDKs), a real cost noted here rather than hidden — code-splitting the voice assistant route is a reasonable follow-up, not done in this task; retrieval quality is sensitive to how a question is phrased (a generic "What loans are available?" scores lower than a scheme-name-specific phrasing) — acceptable given the empirically-verified floor, but a known limitation of small-corpus zero-shot embedding search, not a solved problem.
