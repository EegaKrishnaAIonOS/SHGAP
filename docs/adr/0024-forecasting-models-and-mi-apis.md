# ADR-0024: Forecasting models, backtesting, hotspots/seasonality, and MI APIs

- **Status:** Accepted
- **Date:** 2026-07-23
- **Sprint:** Sprint 3 (T15)

## Context

T15 builds on T14's feature pipeline (ADR-0023): a demand-forecasting model per product × district, a price trend/short-horizon forecast, backtesting with registered error metrics, geo-clustered demand hotspots, descriptive seasonality, and FastAPI endpoints serving all of it. ADR-0008 (Sprint 0) speculatively planned Prophet + XGBoost + Darts for this; T15 is where that plan meets the actual pilot data shape, and it deviates from ADR-0008 in several concrete ways documented below.

## Decision

**Prophet per product+district for demand, one pooled XGBoost regressor for price — not Darts, not per-commodity price models.** `demand_model.py` fits a separate `Prophet(yearly_seasonality=False, weekly_seasonality=True)` model per product (yearly seasonality is off because real history is well under a year — fitting it would just be fitting noise), with `is_weekend`/`is_festival_window` as regressors. `price_model.py` fits a _single_ `xgb.XGBRegressor(enable_categorical=True, tree_method="hist")` pooled across every commodity/market, using commodity/market as native pandas-categorical features, rather than one fragile model per commodity — Agmarknet is snapshot-only (ADR-0023), so any one commodity has far too little real history to fit its own model for a long time; pooling lets one model learn general day-of-week/seasonal pricing patterns from every row while still distinguishing commodities via their categorical features. **Darts was dropped**: it standardizes backtesting/evaluation across model libraries, but Prophet and XGBoost's own native APIs (`model_to_json`/`model_from_json`; `.save_model()`/`.load_model()`) are simpler and less version-fragile than routing both through a third abstraction layer for a 90-day pilot; a small hand-written `backtest.py` (time-ordered split + MAE/RMSE/MAPE) does the same job with less dependency risk.

**Series densification for demand, not left as sparse.** `sales_features` (T14) only has rows for days something actually sold. A day with no sale is a real zero-demand observation, not a missing one — `_densify_daily_series()` reindexes each product to its full observed calendar-day range and fills gaps with `total_quantity=0` before fitting Prophet, so the model can learn "this doesn't sell on Tuesdays" instead of never seeing Tuesday.

**Backtesting is a time-ordered split, not random.** `time_based_split()` sorts by date and holds out the last `max(min_test_rows, test_fraction × len)` rows — this measures genuine forward-prediction accuracy, not interpolation. `compute_metrics()` returns MAE, RMSE, and MAPE; MAPE excludes rows where the actual value is 0 (division-by-zero) and returns `None` (not `0.0` or `NaN`) if every actual in the test window was 0, so a real "can't compute this" state is never silently reported as a suspiciously perfect number.

**A model registry mirrors T14's feature-store manifest convention** (`model_registry.py`): a `manifest.json` keyed by `"{model_type}:{key}"` (e.g. `"demand:<product_id>"`, `"price:pooled"`) records training metadata and backtest scores; each model type serializes itself via its own library's recommended format, never raw pickle.

**Both models decline to train below a minimum-data threshold** (`min_demand_training_days=30` per product, `min_price_training_rows=30` pooled) — returning `None` rather than fitting to noise, consistent with ADR-0022/ADR-0023's "state the real gap, don't fabricate" pattern.

**Hotspots use H3 binning itself as the clustering step — no further clustering algorithm is layered on top.** `hotspots.py` aggregates `sales_features` by its existing resolution-7 H3 cell (already computed in T14) and ranks by `total_amount`. The pilot dataset currently has only 3 real distinct SHG locations (one per pilot district), so any additional clustering (e.g. DBSCAN) on top of H3 binning would trivially just recover those same 3 points — adding it would oversell sophistication the current data can't support, not add real insight.

**Seasonality is purely descriptive, computed independently of either forecasting model.** `seasonality.py` returns historical day-of-week/month averages straight from `sales_features` — always available with no training required, always reflects exactly what happened, and doesn't require translating Prophet's internal seasonal-component objects into a plain JSON shape.

**Endpoints: `/forecast/demand`, `/forecast/price`, `/hotspots`, `/seasonality` are flat, top-level routes** (not nested under `/market-intelligence`), matching the sprint plan's own naming. `POST /market-intelligence/train-models` stays under the existing prefix, alongside T14's `refresh-features`/`feature-status`, since it's pipeline/training control in the same sense those are. A weekly APScheduler job (`training_pipeline_interval_hours`, default 168h) retrains both models automatically, at a much longer interval than T14's 24h feature refresh — retraining is more expensive, and models don't meaningfully change day-to-day the way raw features do.

### Real findings and bugs from verifying against real data

- **The Agmarknet shared demo key is now rate-limited** (`429 Too Many Requests`, no reset headers exposed) — a direct consequence of T14's own heavy pagination testing against a shared public key. Combined with ADR-0023's "snapshot-only, history starts shallow" finding, this means `price_features.parquet` has **0 real rows** in this environment today, well below `min_price_training_rows`. The pooled price model's training/forecast _mechanism_ was verified directly against a synthetic fixture matching Agmarknet's exact schema (state/district/market/commodity/variety/`arrival_date` DD/MM/YYYY/min/max/modal_price) — training, backtesting, and the recursive multi-day forecast all work correctly — but there is honestly no real, currently-trained price model to serve; `/forecast/price` will correctly 404 until enough real price rows accumulate.
- **`pandas.DataFrame.groupby([])` raises `ValueError: No group keys passed!`** — found while designing `price_model.py`'s recursive forecast, which calls `add_lag_features()` with `group_cols=[]` on an already-single-series working dataframe (there's nothing left to group by once filtered to one commodity/market). Fixed by branching `add_lag_features()`: `group_cols=[]` now operates directly on the series via `.shift()`/`.rolling()` instead of going through `groupby()` at all.
- **`demand_model.forecast()` crashed with `KeyError: ['y'] not in index`** on first real run — the future dataframe built for not-yet-observed dates never had a `total_quantity` column, but `_prepare_prophet_frame()` unconditionally renames it to `y`. Fixed by adding a placeholder `total_quantity=0.0` column before preparing the frame (the placeholder value is never used — only `ds` and the regressor columns are passed to `model.predict()`).
- **`seasonality.compute_seasonality()` crashed with `KeyError: 'product_id'`** when called with an empty, columnless `sales_features` frame (the real shape `feature_store.read_features()` returns before the feature pipeline has ever run) — filtering by `product_id` before checking `.empty` assumed the column always exists. Fixed by checking `sales_features.empty` first, before any column access.
- **Direct `uvicorn app.main:app` on Windows hits a pre-existing event-loop quirk unrelated to T15's code**: uvicorn's own `asyncio.run()` creates the default `ProactorEventLoop` _before_ it imports/loads the `app.main` module string — by the time `app/main.py`'s own `WindowsSelectorEventLoopPolicy` guard runs, the loop already exists under the wrong policy, so any endpoint touching the async psycopg repository (T15's `/forecast/demand`, but equally T14's own `/refresh-features`) 500s locally. This is a Windows-dev-only artifact of how uvicorn's CLI entrypoint sequences module loading versus loop creation; Docker/Linux (the real CI/deployment target) creates a compatible loop by default and was used for the actual verification below — all T15 endpoints, including the DB-backed ones, were confirmed working end-to-end inside the built container.

## Alternatives Considered

- **Darts for a unified backtest/serialization layer across Prophet and XGBoost** — rejected; each library's own native serialization (`model_to_json`/`.save_model()`) is simpler and less version-fragile than routing both through a third framework for a single pilot's needs, and a ~40-line hand-written `backtest.py` covers the actual requirement (time-ordered split + MAE/RMSE/MAPE).
- **One XGBoost model per commodity for price** (as ADR-0008 originally implied) — rejected; Agmarknet's snapshot-only ingestion means any single commodity accumulates real history far too slowly to support its own model for a long time. A pooled model with commodity/market as categorical features gets a usable model sooner by sharing signal across commodities.
- **DBSCAN or another clustering algorithm on top of H3 bins for hotspots** — rejected for now; with only 3 real distinct pilot locations, a second clustering pass would just re-discover the same 3 points while looking more sophisticated than the data supports. Revisit once real SHG registrations produce many distinct locations per district.
- **Manual `dropna()` on lag/rolling NaN rows before training the price model** — rejected; XGBoost's native missing-value handling (`enable_categorical=True`, `tree_method="hist"`) already copes with the NaNs that necessarily exist in the first `lag` rows per commodity/market, verified directly in testing, so a manual drop would only discard real, usable rows for no benefit.

## Consequences

- Positive: the full model-training → backtest → registry → forecast path was verified end-to-end against the real 1032-row `sales_features` table (all 6 pilot products meet the 30-day threshold and trained real Prophet models with real backtest MAE/RMSE/MAPE); all new endpoints were exercised against a real Docker container on the shgap-dev network, including the DB-backed ones; three real bugs (not hypothetical edge cases) were found and fixed by actually running the code rather than assuming it worked from having been written correctly.
- Trade-offs: the pooled price model cannot be verified against real data today — real Agmarknet history is both snapshot-only (ADR-0023) and currently rate-limited on the shared demo key, so `price_features.parquet` has 0 rows and `/forecast/price` will 404 until enough real rows accumulate across future scheduled pipeline runs (or a dedicated Agmarknet key is registered, removing the shared-key rate limit). The mechanism itself is proven correct via a synthetic fixture, but "proven correct in principle" and "serving real forecasts" are honestly different things, and this ADR states which one is true today.
- Incidental, unrelated-to-T15 finding recorded for future reference: `tests/categorization/test_service.py::TestSuggest::test_refreshes_category_cache_once_ttl_expires` is flaky (a `cache_ttl_seconds=0` timing race — two calls can land within the same clock tick) — confirmed via repeated isolated runs, pre-existing from T11, not introduced or fixed here.
