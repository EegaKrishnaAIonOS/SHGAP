# ADR-0022: Notification Engine â€” provider abstraction, queue design, and real scope limits

- **Status:** Accepted
- **Date:** 2026-07-21
- **Sprint:** Sprint 2 (T13)

## Context

T13 asks for the notification-service scaffolded in T03 to become real: a template engine, a provider abstraction over SMS/WhatsApp/Voice/Email (ADR-0011 already picked MSG91, WhatsApp Business API, Exotel and Amazon SES), a BullMQ queue with retries and delivery logs (ADR-0005), and domain events wired up for OTP, buyer enquiry, demand increase, price change and tender opportunity.

## Decision

**One `notifications` BullMQ queue, one job per (event, channel).** `POST /notifications/dispatch` (internal, service-to-service â€” no auth, same trust model as ml-services' endpoints: the private network is the boundary, not a bearer token) resolves an event to its default channel list (`channel-routing.ts`), creates one `Notification` row per channel, and enqueues one job per row carrying only `{ notificationId }`. The BullMQ worker (`NotificationsProcessor`) re-reads the row fresh from Postgres before acting, so the database â€” not the queue payload â€” stays the single source of truth for what's actually being sent. Retries are BullMQ's own (`attempts: 3`, exponential backoff); a `failed` event handler only writes the terminal `FAILED` status once BullMQ has actually given up.

**Every channel provider is swappable via a DI-token factory that defaults to a console/dev-stub, exactly mirroring core-api's existing `SMS_PROVIDER`/`ConsoleSmsProvider` pattern from T05.** `notifications.module.ts` binds `Msg91Provider`/`WhatsappBusinessProvider`/`ExotelProvider`/`SesProvider` only when that channel's credentials are configured; otherwise it binds `ConsoleSmsProvider`/`ConsoleWhatsappProvider`/`ConsoleVoiceProvider`/`ConsoleEmailProvider`, which log instead of sending. This means the service is fully runnable, testable and demoable with zero real provider accounts â€” which matters here specifically because, unlike Groq/Sarvam (T10, instant pay-as-you-go API keys), none of MSG91's DLT template registration, a Meta-verified WhatsApp Business account, an Exotel number, or an SES-verified sending domain can realistically be provisioned inside a single task's session. **Honest accounting of what this means for verification:** the full pipeline â€” dispatch â†’ consent check â†’ template render â†’ queue â†’ retry/backoff â†’ provider call â†’ `Notification` row update â€” was verified against real Postgres and real Redis/BullMQ end-to-end (including a real `core-api â†’ notification-service` OTP round trip that ends in a real issued JWT pair). The four _real_ provider implementations were built to each vendor's documented API contract but were not exercised against a live account of any kind â€” Exotel's is the least certain of the four, since its outbound-call flow normally involves an interactively-configured ExoML "applet" in Exotel's own dashboard that there was no account to confirm against; MSG91, WhatsApp Cloud API and SES's request/response shapes are stable and documented independent of any specific account, so those three carry less uncertainty even though none were live-tested either.

**SMS and WhatsApp templates are modeled as template-id/name + params, never freeform text**, because that's what DLT (India, SMS) and Meta's template-approval process (WhatsApp) actually require â€” not a design preference. `templateKey` values in `templates.ts` are stable placeholders for what would be real DLT-registered template IDs / Meta template names in a production account; there's no way to make these "real" without an account to register them against.

**Consent gating uses the existing `Consent` model's latest-row-wins semantics â€” no schema change.** OTP and buyer-enquiry are treated as transactional (never gated: a login code or a business lead reaching someone isn't marketing). Demand-increase/price-change/tender-opportunity are gated behind a granted, non-withdrawn `MARKETING_NOTIFICATIONS` consent record; no record at all is treated as _not_ granted (DPDP's default-deny posture), not granted-by-default. A blocked send is still written to `notifications` with `status: FAILED, failureReason: 'consent_not_granted'` rather than silently dropped â€” DPDP needs a record of what wasn't sent and why, and reusing the existing `FAILED` status avoids a schema migration for what's genuinely a terminal, auditable outcome.

**`requestOtp` now upserts the `User` row before dispatching**, not just `verifyOtp` (which already did). `Notification.userId` is a required FK â€” without this, a brand-new phone number's very first OTP request would have no user row to attach the notification to. The upserted row is `PENDING_VERIFICATION` until `verifyOtp` actually confirms the code, matching the existing status model exactly.

**Only `OTP` has a real, already-existing producer wired end-to-end in this task.** `BUYER_ENQUIRY`'s trigger (a buyer submitting an enquiry) doesn't exist anywhere in the codebase â€” there is no buyer authentication, no `Enquiry` creation endpoint, and no buyer-facing UI wired to one (the catalogue page's "Enquire" i18n string was never connected to a handler even in the T04 wireframe). Building that is a materially different, larger task (buyer identity doesn't exist at all yet) that belongs with the buyer-matching engine (T14-T17), not this one. `DEMAND_INCREASE`/`PRICE_CHANGE`/`TENDER_OPPORTUNITY` depend on forecasting/recommendation logic that also doesn't exist yet (T14-T17). **This is stated plainly rather than papered over**: the notification-service side of all five events is complete, tested, and correct â€” templates render, routing resolves, consent gates correctly, providers are called correctly â€” but only OTP's producer exists today. The other four are ready for their future producers to call `POST /notifications/dispatch` the moment those producers exist.

**No delivery-webhook receivers (DLR callbacks) in this task.** `NotificationStatus.DELIVERED` is modeled in the schema but nothing sets it â€” real delivery confirmation for MSG91/WhatsApp/SES all arrive asynchronously via provider webhooks, and building four signature-verified public webhook endpoints is roughly comparable in size to everything else in this task combined. `SENT` (the provider accepted the request) is as far as this task goes; delivery-webhook receivers are a real, known gap, flagged here rather than left silently unaddressed.

## Alternatives Considered

- **A queue per channel** (four BullMQ queues instead of one) â€” rejected; one queue with a channel field on each job is simpler to operate and reason about for this volume, and BullMQ's per-job `attempts`/backoff already gives per-send retry control without needing separate queues.
- **Building the buyer-enquiry endpoint as part of this task** to give `BUYER_ENQUIRY` a real trigger too â€” rejected once it became clear buyer authentication doesn't exist at all (Buyer has no relation to User anywhere in the schema); that's a genuinely separate, larger piece of work than "wire a domain event," not a small addition.
- **Passing full message content through the BullMQ job payload** instead of just `{ notificationId }` â€” rejected so the database, not an in-flight job, stays authoritative; also keeps job payloads small and makes the queue resilient to a template registry change between enqueue and processing (a re-read always uses current logic).
- **Silently dropping consent-blocked sends instead of writing a FAILED row** â€” rejected; DPDP-relevant decisions need an audit trail, and reusing an existing enum value costs nothing.

## Consequences

- Positive: the full pipeline (dispatch â†’ consent â†’ template â†’ queue â†’ retry â†’ provider â†’ status) is real, tested against real Postgres/Redis, and already proven end-to-end through core-api's actual OTP flow, not just in isolation; every channel degrades gracefully to a working console stub with zero configuration, so the service is immediately useful in dev/CI/demo without any real vendor account; the DLT/Meta template-based design for SMS/WhatsApp avoids building something that would fail regulatory/policy requirements the moment real credentials were added.
- Trade-offs: three of five events (buyer enquiry, demand increase, price change, tender opportunity â€” really four, minus OTP) have no real producer yet and won't fire in practice until later sprint tasks build one; none of the four real provider integrations have been exercised against a live vendor account, so a real deployment should expect to debug at least the Exotel applet step against Exotel's dashboard directly; delivery-status webhooks are unbuilt, so `DELIVERED` will never be set until a follow-up task adds them.
